# B2.0 — Bump infrastructure to Kotlin 1.9.25 + IntelliJ 232

## Goal

Update the plugin infrastructure to a modern Kotlin compiler **without migrating to K2**. Specifically:

- `kotlin.version` 1.3.72 → 1.9.25 (latest 1.x; FE1.0 still default, K2 optional)
- IntelliJ platform 193 + 241 → 232.x (matches the shaded snapshot inside kotlin-compiler:1.9.25)
- `bundled-jars/Kotlin*` source paths: `submodules/Kotlin/idea/…` → `submodules/IntellijCommunity/plugins/kotlin/…`
- IntellijCommunity submodule pin → `idea/232.9921.47` (latest 232 tag)
- `<languageVersion>` 1.3 → 1.9 in `bundled-jars/*` and Nbm

**Out of scope** (deferred to B2.1+ at version 0.7.x):

- K2 Analysis API integration (`buildStandaloneAnalysisAPISession`, `analyze {}` blocks)
- Migrating Nbm call sites from `BindingContext`/`ClassDescriptor` (FE1.0) to `KaSession`/`KaSymbol` (K2)
- Removing `kotlin-compiler.jar` shaded dependency

The Nbm Kotlin code keeps using FE1.0 throughout B2.0 — `BindingContext`, `ClassDescriptor`,
`TopDownAnalyzerFacadeForJVM` continue to work, sourced from kotlin-compiler:1.9.25 which still
ships FE1.0 alongside K2.

## Why 1.9 (not directly K2)

Original B2.1 plan jumped straight from 1.3.72 to 2.3.21. Implementation hit a hard wall: the
shaded `com.intellij.*` classes inside `kotlin-compiler:2.3.21` are **193-era** (a regression in
the JetBrains shading snapshot — see `memory/project_kotlin_compiler_shaded_versions.md`), but K2
standalone analysis-api expects 252-era external platform classes. The two cannot coexist on a
single `com.intellij.*` package namespace; selective stripping turned into a domino: every removed
shaded class drags another into NoClassDefFoundError.

Kotlin 1.9.25 is the sweet spot:

| Layer | Era |
|---|---|
| Shaded `com.intellij.*` inside `kotlin-compiler:1.9.25` | ~232 |
| External `com.jetbrains.intellij.platform:*:232.x` | 232 |
| `submodules/IntellijCommunity@idea/232.9921.47` plugin/kotlin sources | 232 |

All three layers align — no shading conflict. FE1.0 is still the default Kotlin compiler frontend
(K2 stable arrived in 2.0), so nothing in Nbm needs rewriting.

## Target versions

- `kotlin.version` = `1.9.25` (last 1.x)
- `intellij.platform.version` = `232.9921.47` (last 232 tag = IDEA 2023.2.5)
- IntellijCommunity submodule pin = `idea/232.9921.47`
- Kotlin submodule: **no longer used** by bundled-jars; pin can stay at v1.3.72 or be removed
- `<languageVersion>` / `<apiVersion>` = `1.9`
- Plugin version = `0.6.0-SNAPSHOT` (CI cuts `0.6.N` from the new base tag)

## Concrete changes

### 1. Submodule pin

```bash
cd submodules/IntellijCommunity
git fetch origin idea/232.9921.47
git checkout idea/232.9921.47
cd ..
git add submodules/IntellijCommunity
```

The Kotlin submodule keeps its current pin; `bundled-jars/Kotlin*` no longer reference its sources.

### 2. Parent `pom.xml`

- `<version>` 0.5.23-SNAPSHOT → 0.6.0-SNAPSHOT
- `<kotlin.version>` 1.3.72 → 1.9.25
- Add `<intellij.platform.version>232.9921.47</intellij.platform.version>`
- `dependencyManagement`:
  - Bump `core`, `core-impl` from 193.7288.26 → 232.9921.47 (use the property)
  - Bump `code-style`, `code-style-impl`, `project-model`, `concurrency` from 241.194 → 232.9921.47
  - Bump `org.jetbrains.intellij.deps.fastutil:intellij-deps-fastutil` to whatever 232 expects
  - Add any 232-era platform JARs the new shaded snapshot needs transitively (discovered at build time)
- `kotlin-stdlib`, `kotlin-reflect`, `kotlin-script-runtime` automatically follow `${kotlin.version}`

### 3. Children `pom.xml` (all four: Nbm, KotlinIdeCommon, KotlinFormatter, KotlinConverter)

- Parent `<version>` 0.5.23-SNAPSHOT → 0.6.0-SNAPSHOT
- `kotlin-maven-plugin` `<languageVersion>` and `<apiVersion>` 1.3 → 1.9
- Drop legacy compiler args (`-XXLanguage:-ForbidSuperDelegationToAbstractFakeOverride`, `-XXLanguage:-ProhibitVisibilityOfNestedClassifiersFromSupertypesOfCompanion`) — these are 1.3-era flags

### 4. `bundled-jars/Kotlin*/pom.xml` source path rewrites

Replace `${project.basedir}/../../submodules/Kotlin/...` with paths under `${project.basedir}/../../submodules/IntellijCommunity/plugins/kotlin/...`.

Mapping found by inspection of `idea/232.9921.47`:

| Old (`submodules/Kotlin@v1.3.72`) | New (`submodules/IntellijCommunity@idea/232.9921.47`) |
|---|---|
| `idea/ide-common/src/` | `plugins/kotlin/base/fe10/analysis/src/` (and other `*-fe10` paths — discovered by following imports) |
| `idea/formatter/src/` | `plugins/kotlin/formatter/src/` |
| `j2k/src/` + `idea/idea-core/src/` + `idea/ide-common/src/` (KotlinConverter) | `plugins/kotlin/j2k/{old,new}/src/` (FE1.0 = `old`) |

The exact `<include>` lists in `<copy-resources>` will need re-validation: many filenames carried
over (`ReferenceVariantsHelper.kt`, `FuzzyTypeUtils.kt` exist in both); some were renamed or moved
between directories. Iteratively run `mvn install -pl bundled-jars/X` and add/remove entries until
compile succeeds.

The class-strip lists in `bundled-jars/KotlinIdeCommon/pom.xml` (`maven-jar-plugin` `<excludes>`)
need to be re-checked — Nbm's shadow files in `Nbm/src/main/java/org/jetbrains/kotlin/idea/util/`
must still mask the same logical classes. If classes were renamed in 232-era, the strip list and
shadow files must be updated together.

### 5. `Nbm/src/main/java/com/intellij/**` stub adaptation

The 14 stubs that exist for the 0.5 platform mix (193 + 241) may need:

- **Removal** — if 232-era platform JARs already supply the same class with a compatible API
- **Re-add or update** — if 232 introduced new abstract methods we must implement

Procedure: build, run tests, watch for `AbstractMethodError` / `NoSuchMethodError` / `NoClassDefFoundError`
at runtime — each one points at one stub to add or remove. The currently active stubs (per CLAUDE.md):

- `Configurable`, `IndentOptionsEditor`, `CodeStyleSettingsProvider`, `LanguageCodeStyleSettingsProvider`,
  `CodeStyleSettingsCustomizable`, `CodeStyleSettingsService`, `CustomCodeStyleSettingsManager`, `Formatter`,
  `DynamicBundle`, `ConcurrentCollectionFactory`, `ObjectIntHashMap`, `ObjectUtils`, `Extensions`,
  `ContainerUtilRt`, `StringUtil` (last two are generated from submodule via `maven-resources-plugin`)

Most likely outcome: half become unnecessary (232 platform supplies them), half need minor signature tweaks.

### 6. JAR Patches (sources from submodules) — paths

- `Nbm/pom.xml` currently copies `StringUtil.java`, `ContainerUtil.java`, `ContainerUtilRt.java`
  from `submodules/IntellijCommunity/platform/util*/src/com/intellij/...` — paths in 232-tag may
  differ; verify and update.

### 7. `pom.xml` exclusions on `kotlin-compiler:1.9.25`

The current `<exclusions>` in `Nbm/pom.xml` for `kotlin-compiler:1.3.72` exclude packages like
`com/google`, `javaslang`, `org/picocontainer`. The 1.9.25 jar has different bundled packages
(no picocontainer, different shaded transitives). Re-derive the exclusion list from
`unzip -l kotlin-compiler-1.9.25.jar` and what the `Nbm` declared deps already provide.

## Verification checklist (must all pass before merging)

1. `mvn clean test` — all currently-green tests stay green (the same 36 failures + 77 errors that
   were on 0.5 main are acceptable; **regressions** are not). DiagnosticsTest, FormattingTest etc.
   should continue to behave the same way they did before — this stage explicitly does not improve
   them; it only swaps the substrate underneath.
2. `mvn clean package -DskipTests` — `Nbm/target/*.nbm` produced.
3. Install `.nbm` in NetBeans 23+ with the JVM `--add-opens` flags from CLAUDE.md.
4. Open a `.kt` file: no `ExceptionInInitializerError`, no startup crash, syntax highlighting works.
5. Smoke-test FE1.0 features that worked in 0.5: completion, go-to-definition, rename. Behaviour
   should be identical (we did not migrate any call site).
6. `docs/development-plan.md` reflects the B2.0/B2.1 split.
7. `CLAUDE.md` "Key Versions" table updated.
8. PR description links to this plan.

## Known risks

- **Stub blow-up** — adapting 14 stubs to 232 may surface unexpected runtime conflicts. Mitigation:
  iterative build/test; keep a TODO log per stub.
- **`languageVersion=1.3` → `1.9` source incompatibility** — the new sources from
  IntellijCommunity 232 are written for current Kotlin so this is unlikely; the *old* Nbm Kotlin
  sources may use 1.3-era idioms (`@Suppress` keys, deprecated stdlib calls) that 1.9 rejects.
  Fix in place; no semantic rewrite required.
- **NetBeans cluster API mismatch** — 232-era platform expects newer JDK assumptions; verify
  `RELEASE230` (NB 23.0) cluster still resolves. Bump `<netbeans.version>` if forced.
- **JVM `--add-opens` may grow** — 232-era platform might need additional flags beyond the four
  documented in CLAUDE.md. Document any new ones and the user-config recipe.
- **CI cache invalidation** — Maven cache key may need updating after the platform bump.
